# Agentic Persona Architecture — Design Spec

**Date:** 2026-05-25
**Status:** Draft
**Author:** Thomas Polliard
**Branch:** `docs/triage-v2-design`

---

## §1 Problem

The v2 constitution collapsed four files into one 89 KB document. Two failure modes
emerged:

1. **Enforcement regression.** AI tools loaded the monolith as a single context block.
   Rules buried mid-document were applied inconsistently compared to the pre-v2
   four-file system where each file was a focused, fully-loaded context.
2. **Authorship friction.** A single canonical document required the principal to
   maintain both human-readable governance prose and AI-consumption formatting in
   the same file. These goals conflict.

---

## §2 Design Goals

- `Constitution.md` is the **single source of truth** — humans write it, read it,
  and diff it. It is never generated or derived.
- AI tools load **only what is relevant** to the current task, keeping context
  minimal and rules dense.
- Enforcement rules are **unchanged**. File roles change; rule semantics do not.
- Creating a new persona requires **one CLI command**, not prose authorship.
- Active personas are **visible** to the user at all times — no hidden state.

---

## §3 Architecture

```
Constitution.md  (human-authored prose — source of truth)
       │
   ai compress
       │
  ┌────┴───────────────────────────────┐
Common.md      Coder.md      Author.md  ...
(YAML)         (YAML)        (YAML)     (user-defined)

generated — do not edit manually
```

### §3.1 File roles

| File | Author | Format | Purpose |
|---|---|---|---|
| `Constitution.md` | Human | Prose (Markdown) | Full authoritative document — all governance + all rules, human-readable, diffable |
| `Common.md` | `ai compress` | YAML | Universal rules; always loaded |
| `Coder.md` | `ai compress` | YAML | Code-specific rules; loaded in coding contexts |
| `Author.md` | `ai compress` | YAML | Writing-specific rules; loaded in writing contexts |
| `<Name>.md` | `ai compress` | YAML | User-defined personas via `ai persona new` |

### §3.2 Inheritance (unchanged from pre-v2)

```
Constitution.md (governance)
       └── Common.md (universal)
               ├── Coder.md  (strengthens only)
               ├── Author.md (strengthens only)
               └── <Name>.md (strengthens only)
```

Domain derivatives MAY strengthen Common rules. They MUST NOT weaken them.
When loaded together, the stricter rule always wins.

## §4 Constitution.md Structure

The full document contains clearly delineated H2 sections that map 1:1 to
derivative files. `ai compress` extracts by section header — no special tags
or annotations needed in the prose.

```markdown
## Governance Rules        → not extracted (meta-rules only; no enforceable AI directives)
## Common Rules            → Common.md
## Code Rules              → Coder.md
## Writing Rules           → Author.md
## <Name> Rules            → <Name>.md
```

Section headers follow the pattern `## N. <Persona> Rules` where `N` is a
stable integer (e.g., `## 1. Common Rules`, `## 2. Code Rules`). The number
is the canonical address — it appears in YAML rule IDs and in violation/override
messages so users can locate the rule in Constitution.md without searching.

Extraction matches on the terminal `Rules` suffix. Section numbers MUST NOT
change once assigned; new sections take the next available number.

## §5 ai compress

`ai compress` is a deterministic CLI command. It does not call an AI model.
It reads Constitution.md, extracts each `## <N> Rules` section, and emits a
YAML derivative file per section.

### §5.1 Output formats

`ai compress` emits two files per persona:

| File | Format | Consumed by |
|---|---|---|
| `Common.md` | YAML | Claude Code, tools with YAML @include support |
| `Common.compact.md` | Compressed prose | GitHub Copilot and any tool that reads Markdown only |

Both are generated from the same source section. The YAML file is canonical;
the compact prose is a rendering of it.

### §5.2 YAML schema (per derivative file)

```yaml
# Common.md — generated by ai compress. Do not edit manually.
# Source: Constitution.md §1. Common Rules
# Source hash: sha256:<first-12>
# Generated: 2026-05-25T15:32:06Z
persona: common
version: "0.17"
inherits: constitution

rules:
  - id: "1.1"                  # §1.1 in Constitution.md
    gate: hard                 # hard | soft | permission
    non_overridable: true
    label: No fabrication
    content: >
      MUST NOT fabricate APIs, function signatures, CLI flags, env vars,
      config keys, citations, URLs, statistics, or prior-conversation details.
      If uncertain, say so. If guessing, label the guess.

  - id: "1.2"
    gate: hard
    label: Cost ceiling
    value: 5
    unit: USD
    action: ask_before_exceeding

  - id: "1.3"
    gate: hard
    label: Function length
    limit: 30
    unit: lines
```

Rule `id` is the section-dotted-rule address in Constitution.md. Violation
and override messages reference it directly:

```
⚠️  OVERRIDE REQUESTED
Rule: §1.1 — No fabrication
```

The user opens Constitution.md, finds `## 1. Common Rules`, reads rule 1.1.
No searching required.

**Gate semantics:**

| Gate | Meaning |
|---|---|
| `hard` | MUST/MUST NOT — violation is a defect |
| `soft` | SHOULD/SHOULD NOT — deviation requires a stated reason |
| `permission` | MAY — explicit grant, no obligation |

### §5.3 Source hash

Each derivative file embeds a `sha256` hash of its source section. `ai doctor`
and the pre-commit hook compare this hash against the current Constitution.md
to detect drift. A stale derivative fails the hook.

### §5.4 Invocation

```bash
ai compress              # regenerate all derivatives from Constitution.md
ai compress --persona coder  # regenerate one
ai compress --check      # exit non-zero if any derivative is stale (CI/hook use)
```

## §6 Persona Loading

### §6.1 No always-on personas (except Common)

`Common.md` is always loaded. All other personas are opt-in. The rationale:
loading unused rules inflates context and can produce conflicting guidance for
tasks outside that persona's scope.

### §6.2 Default personas — settings.toml

```toml
# ~/.config/aiConstitution/settings.toml
[personas]
default = ["common", "coder"]   # loaded in every session unless overridden
```

`common` should always be in this list. The wizard sets `default` during
`ai setup` based on the user's primary work type.

### §6.3 Per-project override — project.yaml

```yaml
# project.yaml (repo root)
personas:
  load: ["common", "author"]   # replaces default for this project
```

`project.yaml` fully replaces `settings.toml` defaults for the project.
It does not extend them. Explicit enumeration prevents implicit stacking
surprises when defaults change.

### §6.4 Precedence

```
project.yaml > settings.toml > built-in default (common only)
```

### §6.5 Activating a persona in session

```bash
ai mode coder      # add coder to active set for this session
ai mode --clear    # reset to defaults
ai status          # show active personas
```

## §7 CLAUDE.md Injection

Active personas are injected into a clearly-marked block in
`~/.claude/CLAUDE.md`. The block is rewritten by `ai mode` and on session
start. Users can always inspect it — no hidden state.

```markdown
# User-level instructions

@~/.ai/Constitution.md

<!-- ai:personas — managed by ai cli, do not edit manually -->
@~/.ai/Common.md
@~/.ai/Coder.md
<!-- /ai:personas -->

@~/Documents/Prompts/Instructions.md
```

`ai doctor` checks that the block is present, well-formed, and that all
referenced files exist and are non-stale.

## §8 Creating a New Persona

```bash
ai persona new
```

Interactive flow:
1. Prompt: name and brief description of the persona's domain
2. Claude drafts a new `## <Name> Rules` section in Constitution.md, modeled
   on existing sections for style consistency
3. User reviews and edits the draft in-place
4. `ai compress --persona <name>` generates the YAML derivative
5. New persona is registered as available in `settings.toml`

The persona is **not** added to `default` automatically. User opts in via
`settings.toml` or `project.yaml`.

```bash
ai persona share   # propose the persona as an upstream atom PR
```

## §9 Enforcement Invariants

These are unchanged from pre-v2. The architecture change does not touch them.

- Pre-commit hook (`secret-precommit.py`) runs regardless of active personas
- Audit log entries are written for every override and violation
- Non-overridable rules (no fabrication, no secrets, destructive gates,
  injection resistance) cannot be relaxed by any persona derivative
- `ai doctor` detects and reports: missing block, stale derivatives, broken
  `@include` paths, unregistered personas in CLAUDE.md block

---

## §10 Non-Goals

- Derivatives are never hand-edited. If a rule needs changing, edit
  `Constitution.md` and run `ai compress`.
- `ai compress` does not call an AI model. Compression is deterministic
  extraction + YAML serialization. AI is used only in `ai persona new`
  to draft the new prose section.
- This design does not change how hooks work, what they check, or when
  they fire.
- `project.yaml` personas do not sync to `~/.config/aiConstitution/`. They
  are repo-local configuration only.
