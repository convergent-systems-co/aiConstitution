# Plan: ai skills available + ai doctor skills check

**Issues:** #365, #367
**Objective:** Add `ai skills available` to list installable skills from skill-atoms.com, and extend `ai doctor` with a skills-installed check.

## Rationale

| Alternative | Pros | Cons | Verdict |
|---|---|---|---|
| Fetch directory listing via GitHub Contents API | Already have `SkillAtomsBaseURL` pointing there; pattern matches existing `install` code; no new deps | Requires two-level fetch (dir listing → each atom JSON) | Chosen |
| Maintain a static manifest file in skill-atoms repo | Single fetch, no per-file iteration | Out-of-date without CI; adds governance burden | Rejected |
| Do nothing | No work | Users have no discoverability | Rejected |

## Scope

**Files modified:**
- `src/cmd/ai/cmd/skills.go` — add `skillAtomEntry` type, `fetchSkillsDirectory`, `runSkillsAvailable`, `newSkillsAvailableCmd`, register in `newSkillsCmd`
- `src/cmd/ai/cmd/doctor.go` — add `checkInstalledSkills`, call it from `runDoctor`
- `src/cmd/ai/cmd/skills_test.go` — `TestSkillsAvailable_Success`, `TestSkillsAvailable_Empty`, `TestSkillsAvailable_FiltersDeprecated`
- `src/cmd/ai/cmd/doctor_test.go` — `TestDoctorSkillsCheck_NoSkills`, `TestDoctorSkillsCheck_WithSkills`

## Approach

1. **`skills.go` additions:**
   - Add `skillAtomEntry` struct for GitHub Contents API directory entry: `{Name, DownloadURL string}`
   - Add `fetchSkillsDirectory()` that fetches `SkillAtomsBaseURL + "/skills/skill"` and decodes a `[]skillAtomEntry`
   - Add `runSkillsAvailable(cmd, _)` that:
     1. Calls `fetchSkillsDirectory()`
     2. For each `.json` entry, fetches the atom JSON via `DownloadURL`
     3. Filters deprecated/retired lifecycle
     4. Renders a tabwriter table: NAME | VERSION | DESCRIPTION
     5. Emits `(no skills available)` when list is empty after filtering
   - Add `newSkillsAvailableCmd()` wrapping `runSkillsAvailable`
   - Register in `newSkillsCmd()`

2. **`doctor.go` additions:**
   - Add `checkInstalledSkills(w)` that calls `listSkillDirs(skillsManifestDir())`, prints OK or WARN
   - Call it at the end of `runDoctor`

3. **Tests (TDD — written before implementation):**
   - Mock the GitHub Contents API server to return a JSON array of entries
   - Each entry has a `download_url` pointing back to the same or another mock server
   - Verify table output contains name/version/description
   - Verify deprecated entries are skipped
   - Verify `(no skills available)` on empty response
   - Doctor tests: set `AI_ROOT` to empty vs populated skills dir

## Testing Strategy

Tests mock the HTTP server using `httptest.Server` (same pattern as existing install tests). `SkillAtomsBaseURL` is overridden via the existing `setSkillAtomBaseURL` helper.

The `skillAtom` struct already has `lifecycle` missing — need to add it for the filtering requirement. The field is simply added without breaking existing tests (JSON unmarshalling is lenient on unknown fields).

## Risk Assessment

- **Rate limiting:** GitHub Contents API is subject to rate limits; tests use a mock server so no real network calls in tests.
- **Schema drift:** If skill-atoms adds new fields, they are simply ignored.
- **Empty lifecycle field:** Treated as not deprecated (safe default).

## Dependencies

None. All required packages (`text/tabwriter`, existing HTTP machinery) are already imported or available in stdlib.

## Backward Compatibility

Adding a new subcommand and a new doctor check does not break any existing commands or tests.
